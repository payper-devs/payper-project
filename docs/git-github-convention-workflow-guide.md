# Overview

Git 브랜치 및 GitHub에서의 프로젝트 관리 워크플로우에 대해 설명합니다.

# Naming

종류별로 아래와 같은 네이밍 컨벤션을 지킵니다.

| 종류 | 형식 | 예시 |
| --- | --- | --- |
| Issues | [TAG]: Description | [FEATURE]: 로그인 화면 구현 |
| Branch | tag/#:number | feature/#123 |
| Pull Request | [TAG #:issue-number]: Issue Description | [FEATURE #123]: 로그인 화면 구현 |
| Merge Commit | [TAG #:issue-number]: Issue Description (#:pr-number) | [FEATURE #123]: 로그인 화면 구현 (#120) |

# Issues

코드베이스(리포지토리)에 대한 변화는 아무리 사소해도 이슈 생성을 우선합니다. 이슈는 다음과 같은 형식을 지킵니다.

- 이슈는 한 번에 하나의 문제 혹은 기능을 해결합니다.
- 각 이슈가 다룰 범위를 명확하게 설명합니다.
- 이슈는 반드시 Assignees를 지정합니다.
- 이슈는 적당한 Label를 지정합니다.
- 이슈의 프로젝트를 팀 프로젝트로 지정합니다.

## 이슈 템플릿

### 요구 사항

```markdown
## 문제점 및 요구 사항

- 특정 기능을 추가합니다.

## 해결 및 구현 방안

- 기능을 다음과 같이 구현합니다.

## 고려 사항 및 대안 (선택)

- 다른 방식을 선택할 수 있으나, 이런 이유에서 이 방식을 선택합니다.

## 추가 정보 (선택)

- 참고할 예정인 공식문서를 첨부합니다:
```

### 버그 리포트

```markdown
## 설명

버그나 이슈에 대한 간단한 요약을 작성합니다. 어떤 문제가 발생했는지 한두 문장으로 설명해주세요.

## 재현 방법

문제를 재현할 수 있는 구체적인 단계(step)를 순서대로 작성합니다.
- 예:
	1. 홈페이지에 접속한다.
	2. 로그인 버튼을 클릭한다.
	3. 오류 메시지가 표시된다.

## 기대 결과

정상적으로 동작했다면 어떤 결과가 나와야 하는지 작성합니다.
예: 로그인 후 대시보드 화면으로 이동해야 함.

## 실제 결과

실제로 발생한 결과를 구체적으로 작성합니다.
- 예) 로그인 버튼을 눌러도 반응이 없거나, 오류 메시지가 나타남.

## 스크린샷 (선택)

문제를 시각적으로 보여줄 수 있는 스크린샷이 있다면 첨부해주세요. 디버깅에 큰 도움이 됩니다.

## 환경 정보 (선택)

문제가 발생한 운영체제 및 브라우저/디바이스 정보를 작성해주세요.
- 예) Windows, Chrome 11
- 예) iOS, Safari 22

## 추가 정보 (선택)

문제를 파악하는 데 도움이 될 수 있는 기타 참고 정보를 작성해주세요.
- 예) 콘솔 로그, 에러 메시지, API 응답 값 등
```

## 이슈 태그

| 종류 | 설명 |
| --- | --- |
| [FEATURE] | 새 기능 혹은 요구 사항입니다. |
| [REFACTOR] | 기존 코드 및 아키텍처를 개선합니다. |
| [BUG] | 뭐가 잘 안 돼요. |
| [DEVOPS] | 배포, CI/CD 및 프로젝트 자동화 관련 이슈입니다. |
| [DOCS] | 프로젝트 문서를 수정 및 추가합니다. |
| [TEST] | 테스트 생성 및 수정에 대한 이슈입니다. |
| [CHORE] | 프로젝트에 직접적으로 영향을 주지 않는 기타 업무입니다. |

# Branch

이슈가 생성되면 GitHub를 통해서 브렌치를 생성합니다. 브렌치 이름은 이슈 태그와 이슈 넘버를 조합하여 생성합니다:

```markdown
tag/#issue-number
ex) feature/#123
```

브렌치가 만들어지고 나면 해당 이슈에서의 수정은 모두 해당 브렌치에서 진행합니다. 보통 브렌치 생성 버튼을 누르면 다음과 같이 로컬에서 체크아웃할 수 있는 커멘드 라인을 복사할 수 있습니다.

```bash
git fetch origin
git checkout feature/#141
```

이슈 브렌치 내에서의 커밋 네이밍은 다음과 같이 작성합니다.

```bash
tag: Description.
ex) feature: LogInPage.vue 생성.
```

# Pull Request

이슈에 대한 브렌치에 작업이 끝나면, 해당 원격 브렌치로 푸시하고 풀리퀘스트(Pull Request)를 만듭니다.

- 풀리퀘스트는 현재 변경 사항에 대해 전혀 모르는 사람을 위해 설명한다는 생각으로 내용을 작성합니다.

## PR Template

```markdown
## 관련 이슈

- #number (예: - #123)

## 변경 사항

- 이번 커밋 또는 PR에서 어떤 변경이 이루어졌는지 정리합니다.

## 고려 사항 및 주의 사항 (선택)

- 기능 동작에 영향을 줄 수 있는 사항이나, 리뷰 시 특히 확인이 필요한 내용을 기재합니다.
- 예: “기존 기능과의 충돌 가능성 있음”, “의존성 패키지 버전 업그레이드 포함됨” 등

## 추가 정보 (선택)

- 테스트 방법, 참고 자료, 디자인 문서 링크 등 추가적인 정보가 있을 경우 기재합니다.
```

# Merge

PR이 머지되기 위해서는 다음 조건을 충족해야 합니다:

1. CI 자동 테스트를 통과합니다(통과시킬려고 테스트 지우고 그러면 안됩니다).
2. 최소 1명에게 코드 리뷰를 승인 받습니다.

브랜치 머지는 다음과 같이 진행합니다.

- 각 이슈는 `develop` 브렌치에 Squash and Merge 합니다.
- `develop` 브렌치의 통합 테스트가 완료되면 `main` 브렌치로 Rebase and Merge 합니다.

`develop`에 Squash and Merge 할 때는 다음과 같은 네이밍 컨벤션을 지킵니다:

```
[TAG #:issue-number]: Issue Description (#:pr-number)

ex) [FEATURE #123]: 로그인 화면 구현 (#120)
```
